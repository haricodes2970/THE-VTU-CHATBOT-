## Problems Solved

### 1. Scrape State Lost on Redeploy
**Problem:** Every `git push` triggers a Render redeploy which wipes the ephemeral
filesystem — pending queue and processed URL lists (JSON files) were deleted every time.

**Fix:** Added `ScrapeState` PostgreSQL table (key-value store). Pipeline now writes
queue state to DB first, JSON files as fallback. Survives redeploys.

---

### 2. discover() Timing Out
**Problem:** `_get(page_url)` with 2s sleep × 10 pages = 20s sleeping alone, plus
fetch time. Exceeded Render's 30s HTTP timeout.

**Fix:**
- Added `start_page` and `max_pages` params → call twice (pages 1-5, then 6-10)
- Reduced listing page delay from 2.0s to 0.5s

---

### 3. process-next Timing Out (66s+ per post)
**Problem:** `_get_fast` called `_get(delay=0)` which still had `@retry(attempts=3,
wait_exponential)`. On slow VTU server: 3 attempts × 20s timeout + 2s + 4s waits = 66s+.

**Fix:** `_get_fast` now directly calls `self.session.get()` with no retry decorator,
no delay. Completely bypasses the retry logic.

---

### 4. PDF Parsing OOM (230MB per PDF)
**Problem:** pdfplumber + pytesseract workflow: extract_tables() + extract_text()
(double pdfplumber pass), then OCR fallback converting each page to 200 DPI images
(~170MB per page). Crashed 512MB Render instance.

**Fix (Phase 1):** `skip_pdf=True` mode — use WordPress post body HTML text instead.
Fast (~7s per post vs 60s+), no memory spike.

**Fix (Phase 2):** Replaced pdfplumber + PyPDF2 + pytesseract + Pillow entirely with
`pypdf==4.2.0`. Uses ~15MB RAM. Also removed `tesseract-ocr` from Dockerfile.

**Discovery:** VTU timetable PDFs are scanned images, not text-based. pypdf cannot
extract text from them (returns empty). OCR is required for actual date extraction
but OOMs on 512MB. Conclusion: skip_pdf=True with synthetic metadata is the correct
approach for Render free tier.

---

### 5. Startup OOM (502 on All Requests)
**Problem:** `_load_model()` pre-warm call in `lifespan()` loaded the 80MB ONNX
fastembed model at startup, pushing memory over 512MB → every request got 502.

**Fix:** Removed pre-warm from startup. Model loads lazily on first embed call.
Also baked the model download into the Docker image (`RUN python -c "from fastembed..."`)
so it doesn't download at runtime.

---

### 6. embed-pending Infinite Loop (remaining=96 forever)
**Problem:** After embedding a circular, `is_indexed=True` was never persisting to the
database. Every call to embed-pending picked the same circular again. Ran 120+ times
with `remaining=96, embedded=1` every single iteration.

**Root cause:** SQLAlchemy ORM object tracking silently failed to persist the update.
`circular.is_indexed = True; db.commit()` executed without error but the DB row
was never updated.

**Fix:** Replaced ORM assignment with explicit raw SQL:
```sql
UPDATE circulars SET is_indexed = TRUE WHERE id = :id
```
Also added `/admin/reset-indexed` endpoint (bulk SQL UPDATE) for clean re-embed runs.

---

### 7. Title Always "VTU Timetable"
**Problem:** `soup.find("h1", class_=re.compile("entry-title"))` didn't match VTU's
WordPress theme. All circulars saved with title "VTU Timetable".

**Fix:** Multi-selector fallback:
1. `h1` with class matching `entry-title|post-title`
2. Any `h1`
3. `<title>` tag, split on `–` or `|`

---

### 8. save_circular Never Updating Content
**Problem:** `CircularService.save_circular()` was fully idempotent — if a circular
with the same PDF URL already existed, it returned the existing record unchanged.
Re-scraping with better content had no effect.

**Fix:** If new content is longer than existing content, update the record and reset
`is_indexed=False` so it gets re-embedded with the richer content.

---

### 9. Chatbot Saying "I couldn't find that"
**Problem (Part 1):** System prompt expected actual date patterns (DD/MM/YYYY, FN/AN
session markers) from PDF text. Since we use synthetic metadata content with no dates,
LLM always returned `NOT_FOUND`.

**Fix:** Rewrote system prompt to work with what we actually have — timetable metadata
(scheme, semester, session, PDF link). LLM now shows the timetable info and PDF link.

**Problem (Part 2):** Even with fixed prompt, retriever found relevant chunks but LLM
returned NOT_FOUND. The fallback showed a generic PDF link, ignoring the actual retrieved
PDF links sitting in the sources list.

**Fix:** When NOT_FOUND but retrieved chunks exist, build the answer directly from the
chunk metadata — show each unique PDF link with its scheme/semester/session label.

## Architecture Changes

### New Admin Endpoints
| Endpoint | Purpose |
|----------|---------|
| `POST /admin/discover?start_page=N&max_pages=N&force=bool` | Phase 1: queue post URLs from VTU listing pages |
| `POST /admin/process-next?batch=N&embed=bool&skip_pdf=bool` | Phase 2: scrape posts, save to DB |
| `POST /admin/embed-pending?batch=N` | Phase 3: embed unindexed circulars into Pinecone |
| `POST /admin/reset-indexed` | Bulk-reset all is_indexed=False for clean re-embed |
| `GET /admin/scrape-stats` | Counts: circulars, vectors, queue state |
| `GET /admin/debug-state` | Raw ScrapeState DB table contents |
| `GET /admin/test-scraper` | Fetch one VTU page, return found links (no DB writes) |

### New DB Table
```python
class ScrapeState(Base):
    __tablename__ = "scrape_state"
    key: Mapped[str] = mapped_column(String(50), primary_key=True)
    data: Mapped[str] = mapped_column(Text, default="[]")
    updated_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow)
```

### chunker.py — New Method
`chunk_timetable(text, metadata)` — each line containing a date (DD/MM/YYYY) becomes
its own Pinecone chunk for precise per-subject retrieval. Also adds a full-text chunk.

### Known Limitation
VTU timetable PDFs are scanned images. pypdf cannot extract text from them.
Actual per-subject exam dates (e.g. "Mathematics: 15/01/2026 FN") are only inside
the PDF images and cannot be extracted without OCR. OCR (tesseract) OOMs on 512MB.

**Workaround:** Chatbot directs students to the correct timetable PDF link. Students
check the PDF for specific subject dates.

**To get actual dates:** Would require either a paid Render instance (more RAM for
OCR) or a cloud OCR service (Google Vision, AWS Textract).

## How to Re-Run the Full Pipeline (Future Reference)

```bash
# 1. Clear scrape state (keeps circulars in DB)
POST /api/v1/admin/clear-state

# 2. Discover post URLs (call twice for all 10 pages)
POST /api/v1/admin/discover?force=true&start_page=1&max_pages=5
POST /api/v1/admin/discover?start_page=6&max_pages=5

# 3. Process posts into DB (repeat until remaining=0)
POST /api/v1/admin/process-next?batch=5&embed=false&skip_pdf=true

# 4. Reset embed flags
POST /api/v1/admin/reset-indexed

# 5. Embed into Pinecone (repeat until remaining_unindexed=0)
POST /api/v1/admin/embed-pending?batch=3
```

All admin endpoints require header: `X-Admin-Key: <SECRET_KEY>`
